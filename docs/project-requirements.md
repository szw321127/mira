# RedNote 当前项目需求文档

更新日期：2026-06-12

## 1. 背景

小红书创作者、运营人员和内容团队经常只有一个模糊选题，但缺少稳定结构、可比较方向和可直接发布的图文内容。RedNote 需要帮助用户在一个连续工作台里完成从“输入想法”到“得到可复制发布内容”的主要决策，并保留创作过程，降低重新开始的成本。

当前项目已经形成四部分：

- 用户侧创作工作台。
- 后端 API 与数据持久化。
- 可复用的 `@rednote/agent` 内容分析能力。
- 管理后台，用于配置模型、内容来源和管理项目。

本需求文档根据当前设计文档反推，描述当前系统已经承载或必须满足的产品需求。

## 2. 目标

### 2.1 产品目标

- 帮用户把一句话想法转为多组可选择、可编辑的大纲。
- 支持用户保留旧大纲并换一批，比较不同创作方向。
- 支持用户选择大纲后生成最终可直接复制发布的小红书图文内容。
- 支持生成封面图或封面图提示，帮助用户形成完整发布素材。
- 支持导入小红书参考内容，用于提炼结构、钩子、标签和内容模式。
- 支持完整保留当前工作状态并恢复。
- 支持后台配置模型和内容来源，避免把供应商信息写死在代码里。

### 2.2 非目标

当前阶段不要求：

- 直接内置小红书官方或私有采集 SDK。
- 直接发布到小红书。
- 完整团队权限体系。
- 多租户计费。
- 复杂异步任务队列。
- 生产级 PostgreSQL 部署方案。

这些能力可作为后续扩展。

## 3. 用户与角色

### 3.1 普通创作者

典型用户：

- 小红书个人博主。
- 需要快速起稿的内容创作者。
- 有一个主题或灵感，但不知道怎么组织内容。

核心诉求：

- 快速获得多个不同表达方向。
- 能编辑大纲，而不是被动接受 AI 输出。
- 能得到可直接复制发布的标题、正文、标签和封面提示。
- 能保存和恢复工作状态。

### 3.2 内容运营人员

典型用户：

- 品牌内容运营。
- 账号矩阵运营。
- 商业内容策划。

核心诉求：

- 能参考已有小红书笔记或账号内容。
- 能提炼内容模式、互动信号和标签方向。
- 能根据参考内容生成更贴近平台语境的发布包。
- 能保存多次生成结果，便于比较和交付。

### 3.3 系统管理员

典型用户：

- 项目维护者。
- 运营负责人。
- 技术管理员。

核心诉求：

- 能登录后台。
- 能配置文本模型和图片模型。
- 能配置内容来源 provider。
- 能管理项目、任务和风险状态。
- 能查看审计日志。
- 能修改管理员资料和密码。

## 4. 总体范围

### 4.1 用户侧范围

- 账号注册、登录、Google 登录和 Demo 登录。
- 创作对话创建、切换、删除。
- 想法输入。
- 大纲生成、追加批次、编辑、选择。
- 小红书参考导入。
- 发布包生成、修复、复制。
- 封面图生成。
- 手动保存草稿。
- 自动保存和恢复工作状态。

### 4.2 后端范围

- 统一 API 返回格式。
- 用户认证和管理员认证。
- 创作数据持久化。
- 模型服务调用。
- 小红书分析和内容来源导入。
- 管理后台配置。
- 审计日志。

### 4.3 管理后台范围

- 管理员登录。
- 管理员信息维护。
- 项目总览、项目列表、任务看板。
- 模型配置。
- 内容来源配置。
- 审计日志展示。

## 5. 功能需求

### REQ-AUTH-001 用户注册

系统应允许用户使用账号、展示名和密码注册。

验收标准：

- 用户提交账号、展示名、密码后可创建用户。
- 已存在账号不能重复注册。
- 密码必须保存为哈希，不保存明文。
- 注册成功后返回 access token 和用户信息。

### REQ-AUTH-002 用户账号密码登录

系统应允许用户使用账号和密码登录。

验收标准：

- 用户输入正确账号和密码后获得 access token。
- 账号不存在、密码错误或账号无密码时返回统一错误。
- 登录后前端保存 session 并进入工作台。

### REQ-AUTH-003 Google 邮箱登录

系统应支持 Google Credential 登录。

验收标准：

- 前端通过 `NEXT_PUBLIC_GOOGLE_CLIENT_ID` 初始化 Google 登录按钮。
- 后端校验 Google credential。
- Google 邮箱未注册时自动创建用户。
- Google 邮箱已注册普通账号时可绑定 `googleSub` 并登录。
- 未验证邮箱不能登录。

### REQ-AUTH-004 Demo 登录

系统应支持 Demo 登录用于本地或非生产环境体验。

验收标准：

- 非生产环境默认允许 Demo 登录。
- 生产环境只有 `ENABLE_DEMO_LOGIN=true` 时允许 Demo 登录。
- Demo 登录会创建或更新默认演示用户。

### REQ-AUTH-005 用户会话恢复

系统应支持用户刷新页面后恢复登录态。

验收标准：

- 前端从 localStorage 读取用户 session。
- 前端调用 `/auth/me` 校验 token。
- token 有效时恢复工作台。
- token 无效时清除 session 并展示登录界面。

### REQ-CONV-001 创建创作对话

系统应允许用户创建新的创作对话。

验收标准：

- 创建对话时必须保存用户输入的 topic。
- 可选保存 title。
- 创建后返回完整 conversation 状态。
- 对话归属于当前登录用户。

### REQ-CONV-002 查看和切换对话

系统应允许用户查看自己的对话列表并切换对话。

验收标准：

- 对话列表按更新时间倒序。
- 列表包含标题、topic、最近打开时间、大纲批次数、草稿数、保存草稿数。
- 用户只能查看自己的对话。
- 打开对话时更新 `lastOpenedAt`。

### REQ-CONV-003 删除对话

系统应允许用户删除自己的对话。

验收标准：

- 删除对话会级联删除大纲、草稿、快照、保存草稿和参考来源。
- 用户不能删除不属于自己的对话。
- 删除成功返回 `{ ok: true }`。

### REQ-IDEA-001 输入创作想法

系统应提供一个主输入区域，让用户输入创作想法。

验收标准：

- 输入内容会进入工作台状态。
- 自动保存时同步到 conversation topic。
- 空想法不得触发有效生成。

### REQ-OUTLINE-001 生成一批大纲

系统应根据用户想法生成一批大纲。

验收标准：

- 一次生成 3 个大纲。
- 每个大纲包含标题、钩子、标签、语气和 5 个要点。
- 大纲语气应在 `guide`、`story`、`checklist` 中。
- 生成失败时向用户展示错误状态。

### REQ-OUTLINE-002 换一批并保留旧大纲

系统应支持用户换一批大纲，并保留之前生成的大纲。

验收标准：

- 新大纲追加为新的 batch。
- 旧 batch 不被清空。
- 大纲按 batch 分组展示。
- 新 batch 生成后可默认选中其中一个大纲。

### REQ-OUTLINE-003 编辑大纲

系统应允许用户编辑大纲内容。

验收标准：

- 用户可编辑标题、钩子、标签、语气和要点。
- 要点不能为空。
- 编辑已持久化的大纲时同步到后端。
- 大纲更新后相关草稿标记为过期或 stale。

### REQ-OUTLINE-004 选择大纲

系统应允许用户选择一个大纲作为生成依据。

验收标准：

- 用户选择后更新当前 `selectedOutlineId`。
- 选择状态自动保存。
- 用户确认生成时使用当前选中大纲。

### REQ-DRAFT-001 生成图文草稿

系统应根据用户想法和选中大纲生成可发布图文草稿。

验收标准：

- 未选择大纲时不能生成草稿。
- 草稿包含标题、封面短句、正文、标签和图片提示词。
- 草稿内容应面向小红书发布，不应只是给 AI 的提示或创作建议。
- 生成新草稿时旧草稿标记为 stale。

### REQ-DRAFT-002 编辑图文草稿

系统应允许用户编辑生成后的图文草稿。

验收标准：

- 用户可编辑标题、封面短句、正文段落、标签和图片提示词。
- 已持久化草稿的编辑应防抖同步到后端。
- 本地 XHS 发布包草稿应随工作台快照保存。

### REQ-DRAFT-003 手动保存草稿

系统应允许用户手动保存当前草稿。

验收标准：

- 可保存真实后端草稿。
- 可保存本地 XHS 发布包草稿快照。
- 完全重复的草稿不应在前端重复展示。
- 保存记录包含保存时间。

### REQ-DRAFT-004 复制发布内容

系统应允许用户复制最终发布内容。

验收标准：

- 用户可以复制标题、正文、标签或完整发布文本。
- 复制内容应是可直接粘贴到小红书的文本。

### REQ-IMAGE-001 生成封面图

系统应根据草稿标题、封面短句、标签和图片提示词生成封面图。

验收标准：

- 用户可点击生成封面图。
- 生成期间显示 `generating` 状态。
- 成功后展示图片。
- 失败后展示错误信息。
- 支持 base64 图片和 URL 图片两种模型返回。

### REQ-AUTOSAVE-001 自动保存工作状态

系统应自动保存用户当前工作状态。

验收标准：

- 用户已登录且存在 conversationId 时启动自动保存。
- 正在生成或新建对话时不触发自动保存。
- 工作状态变化后防抖保存。
- 保存内容包括想法、大纲、选中大纲、草稿、参考来源、状态消息和最近工作流结果。
- 自动保存状态在界面中可见。

### REQ-AUTOSAVE-002 恢复工作状态

系统应支持完整恢复历史工作状态。

验收标准：

- 打开对话时优先从最新 snapshot 恢复。
- 没有 snapshot 时，从后端 conversation aggregate 恢复可用状态。
- 参考来源应从 `XhsReference` 恢复。
- 恢复后用户可继续编辑、生成和保存。

### REQ-XHS-001 导入小红书单篇笔记

系统应允许用户通过 URL 或 noteId 导入小红书笔记作为参考。

验收标准：

- 用户可选择笔记导入模式。
- 请求发送到后端 `/xhs-analysis/posts/import`。
- 后端调用当前配置的内容来源 provider。
- 导入结果归一化为标题、正文、作者、图片、指标、标签等分析输入。
- 若提供 conversationId，导入结果保存为 `XhsReference`。

### REQ-XHS-002 导入小红书账号

系统应允许用户通过账号主页 URL 或 userId 导入账号作为参考。

验收标准：

- 用户可选择账号导入模式。
- 请求发送到后端 `/xhs-analysis/accounts/import`。
- 后端调用当前配置的内容来源 provider。
- 导入结果归一化为账号信息和代表性笔记。
- 若提供 conversationId，导入结果保存为 `XhsReference`。

### REQ-XHS-003 管理参考来源

系统应允许用户查看和删除当前对话中的参考来源。

验收标准：

- 打开对话时加载保存过的参考来源。
- 参考来源包含账号或笔记分析结果。
- 删除参考来源会更新前端状态。
- 若参考来源有后端记录，删除时调用后端删除接口。

### REQ-XHS-004 基于参考来源生成创作 brief

系统应能根据导入的账号和笔记生成创作 brief。

验收标准：

- 无参考来源时可跳过 brief。
- 有参考来源时调用 `/xhs-analysis/generation-brief`。
- brief 应包含 prompt additions、推荐结构和 source patterns。
- 参考内容只用于提炼模式，不应要求照搬原内容。

### REQ-XHS-005 基于参考来源生成候选大纲

系统应能根据想法、参考 brief 和目标人群生成小红书候选大纲。

验收标准：

- 接口返回多个候选大纲。
- 候选大纲包含标题、策略、适用人群、页数估计、选择理由和具体 outline。
- 前端将候选映射为工作台大纲。
- 生成的大纲以 `xhs:` 本地 ID 标识。

### REQ-XHS-006 生成可发布图文包

系统应能基于用户想法、选中大纲和参考来源生成小红书图文发布包。

验收标准：

- 发布包包含标题候选、分页内容、正文、标签、图片提示包和发布检查项。
- 发布包 pages 保持 4 到 7 页。
- 每页包含页码、角色、标题、正文、图片提示和设计备注。
- 生成结果应可直接复制发布。

### REQ-XHS-007 审查和修复发布包

系统应审查发布包是否满足发布要求，并支持修复。

验收标准：

- 发布包审查返回 ready、score、blockers、warnings、passed checks 和 repair actions。
- 若发布包未 ready，用户可触发修复。
- 修复后再次审查并返回新结果。
- 修复结果不得只返回建议，应返回完整发布包。

### REQ-ADMIN-AUTH-001 管理员初始化

系统应在没有管理员时自动创建初始管理员。

验收标准：

- 默认账号为 `admin`。
- 默认密码为 `Rednote@123456`。
- 可通过环境变量覆盖初始账号和密码。
- 生产环境未配置初始密码时应阻止启动。

### REQ-ADMIN-AUTH-002 管理员登录

系统应支持管理员登录后台。

验收标准：

- 管理员输入账号密码后获得 admin access token。
- 后台前端保存 token。
- token 有效时进入管理工作区。
- token 无效或 401 时退出到登录页。

### REQ-ADMIN-AUTH-003 管理员资料和密码

系统应支持管理员查看和修改资料及密码。

验收标准：

- 管理员可查看账号、显示名、创建时间、更新时间和最近登录时间。
- 管理员可修改显示名。
- 管理员修改密码时必须提供当前密码。
- 新密码必须满足长度要求。

### REQ-ADMIN-PROJ-001 项目总览

管理后台应展示项目管理总览。

验收标准：

- 展示项目总数、活跃项目数、风险项目数、平均进度和已完成任务数。
- 展示通知、项目列表、风险队列和任务列表。
- 数据来自后端 `/admin/projects/dashboard`。

### REQ-ADMIN-PROJ-002 创建项目

管理后台应允许创建项目。

验收标准：

- 项目必须包含名称和负责人。
- 可配置 key、状态、优先级、预算、截止时间、进度和团队。
- 项目 key 唯一。
- 创建成功后写入审计日志。

### REQ-ADMIN-TASK-001 管理任务

管理后台应允许创建、更新和删除任务。

验收标准：

- 任务必须包含名称、负责人和截止时间。
- 任务可选关联项目。
- 任务 key 唯一。
- 任务状态支持 `待开始`、`推进中`、`验收中`、`已完成`。
- 创建、更新、删除任务写入审计日志。

### REQ-ADMIN-MODEL-001 配置文本模型

管理后台应允许配置文本模型。

验收标准：

- 可配置 `baseUrl` 和 `modelName`。
- 可添加多个 API key。
- 可启用、禁用、更新和删除 API key。
- API key 后端加密保存，前端只显示脱敏预览。
- 可测试模型连接。

### REQ-ADMIN-MODEL-002 配置图片模型

管理后台应允许配置图片模型。

验收标准：

- 可配置 `baseUrl` 和 `modelName`。
- 可添加多个 API key。
- 可启用、禁用、更新和删除 API key。
- API key 后端加密保存，前端只显示脱敏预览。
- 可测试图片生成接口连接。

### REQ-ADMIN-PROVIDER-001 配置内容来源

管理后台应允许配置小红书内容来源 provider。

验收标准：

- 支持 `tikhub` 和 `custom` 两类 provider。
- 可配置名称、baseUrl、启用状态、限流值和合规说明。
- 每类 provider 可添加多个 API key。
- 可启用、禁用、更新和删除 API key。
- API key 后端加密保存，前端只显示脱敏预览。

### REQ-ADMIN-AUDIT-001 审计日志

管理后台应展示关键操作审计日志。

验收标准：

- 模型配置变更写入审计日志。
- 内容来源配置变更写入审计日志。
- 项目和任务变更写入审计日志。
- 管理后台可按最新时间查看审计日志。

### REQ-API-001 统一响应格式

所有后端 HTTP 接口应统一响应结构。

验收标准：

- 成功响应为 `{ code: 0, data, msg: "ok" }`。
- 失败响应为 `{ code: httpStatus, data: null, msg }`。
- 前端请求库校验 envelope。
- envelope 异常时前端抛出接口格式错误。

### REQ-API-002 前端请求错误统一处理

用户侧前端应提供类似 axios interceptor 的请求封装。

验收标准：

- 请求前可通过 request interceptor 修改 config。
- 响应后可通过 response interceptor 处理 envelope。
- 错误可通过 error interceptor 统一处理。
- API 错误统一包装为 `ApiError`。

### REQ-CONFIG-001 本地开发配置

系统应支持本地 monorepo 开发。

验收标准：

- 根目录 `pnpm dev` 可同时启动 backend、web、admin。
- backend 默认端口为 3001。
- web 默认端口为 3000。
- admin 默认端口为 3002。
- SQLite 默认数据库可通过 `DATABASE_URL` 配置。

## 6. 非功能需求

### NFR-SEC-001 密码和密钥安全

系统不得明文存储用户密码、管理员密码、模型 API key 和内容来源 API key。

验收标准：

- 用户和管理员密码使用 Argon2 哈希。
- API key 使用 AES-256-GCM 加密。
- 前端只显示 API key 脱敏预览。

### NFR-SEC-002 权限隔离

系统应区分用户端和管理端权限。

验收标准：

- 用户端接口使用用户 JWT。
- 管理端接口使用 admin JWT。
- 管理员 JWT 包含 `scope: "admin"`。
- 用户不能访问管理接口。
- 管理员 token 失效时后台退出登录。

### NFR-DATA-001 数据归属隔离

用户只能访问自己的创作数据。

验收标准：

- Conversation 查询必须带 userId。
- Outline、PostDraft、Snapshot 和 XhsReference 操作必须校验归属。
- 无权访问时返回 Not Found 或等价错误。

### NFR-UX-001 工作台应降低视觉和认知负担

用户侧界面应像内容编辑工位，而不是营销页或通用聊天框。

验收标准：

- 第一屏直接进入创作工作流。
- 大纲、参考来源和草稿按工作流组织。
- 按钮不应重复堆叠造成视觉过载。
- 关键动作使用清晰图标和 tooltip。
- 中文长文本不应溢出容器。

### NFR-UX-002 后台应高效可扫读

管理后台应适合反复操作和扫描。

验收标准：

- 使用 Ant Design 组件。
- 侧边栏可收起。
- PC 展开/折叠按钮位于 header。
- 只滚动 main 区域。
- header 高度保持紧凑。

### NFR-COMPAT-001 模型服务兼容性

模型服务应采用 OpenAI-compatible 协议。

验收标准：

- 文本模型调用 `chat/completions`。
- 图片模型调用 `images/generations`。
- baseUrl 由后台配置。
- modelName 由后台配置。
- apiKey 由后台配置。

### NFR-CONTENT-001 内容来源合规边界

内容来源应以用户授权导入或自行导入为边界。

验收标准：

- 后台内容来源配置包含合规说明。
- `custom` provider 不内置在主 backend。
- 外部 provider 通过标准接口接入。
- 文档中说明自建连接器需要账号授权、Cookie 加密和限流。

### NFR-TEST-001 基础质量门禁

项目应提供按包测试和构建命令。

验收标准：

- 根目录提供 `pnpm test` 和 `pnpm build`。
- 各包提供独立 test/build 脚本。
- backend 测试覆盖核心 service。
- frontend/admin 测试覆盖关键设计结构。

## 9. 验收总清单

### 用户侧

- 用户可以注册并登录。
- 用户可以使用 Google 邮箱登录，未注册时自动注册。
- 用户可以创建、切换、删除对话。
- 用户可以输入想法并生成 3 个大纲。
- 用户可以换一批大纲，旧大纲仍保留。
- 用户可以编辑和选择大纲。
- 用户可以基于大纲生成可发布图文内容。
- 用户可以编辑、保存、复制草稿。
- 用户可以生成封面图。
- 用户可以导入小红书笔记或账号参考。
- 用户工作状态可以自动保存并恢复。

### 后端

- 所有接口返回 `{ code, data, msg }`。
- 用户端接口鉴权生效。
- 管理端接口鉴权生效。
- Prisma 数据模型覆盖当前业务对象。
- 文本和图片生成从后台模型配置读取。
- 内容来源导入从后台 provider 配置读取。
- XHS 参考来源可保存、查询、删除。

### 管理后台

- 管理员可登录、退出、修改资料和密码。
- 初始管理员可自动创建。
- 管理员可查看项目总览。
- 管理员可创建项目。
- 管理员可创建、更新、删除任务。
- 管理员可配置文本模型和图片模型。
- 管理员可配置多个模型 API key。
- 管理员可配置内容来源 provider。
- 管理员可配置多个内容来源 API key。
- 管理员可查看审计日志。
