# Design: XHS Research Driven Outlines

## Problem Statement

The current creator flow can turn one idea into outline candidates, but the
result can still feel like generic AI generation. The desired product behavior is
more specific: the user enters one idea, RedNote breaks that idea into search
keywords, searches Xiaohongshu for popular notes, analyzes those notes' hooks and
outline structures, then generates outline candidates based on that research.

The user should not have to operate a search console. Research is supporting
evidence for better outlines, not a second workspace.

## What Makes This Cool

The product moment should be: "I typed one rough thought, and the app came back
with directions that feel native to Xiaohongshu right now." The visible output is
still a compact creator workbench, but the generated outlines carry platform
signals: hook patterns, structures, tags, avoid patterns, and source-backed
angle choices.

## Verified Current State

- `packages/backend/src/xhs-analysis/xhs-analysis.controller.ts` exposes manual
  analysis/import/generation endpoints, including `/xhs-analysis/outlines`, but
  has no research-backed endpoint.
- `packages/backend/src/xhs-analysis/xhs-analysis.service.ts` delegates to
  `@rednote/agent/xhs-analysis` and already resolves configured content
  providers for post/account import.
- `packages/backend/prisma/schema.prisma` has `Conversation`, `OutlineBatch`,
  `Outline`, and `XhsReference`, but no durable research run model.
- `packages/agent/src/xhs-analysis/index.ts` already has pure helpers for
  imported post normalization, post/account analysis, generation briefs, outline
  candidates, and publish package audit.
- `docs/xhs-connector-analysis.md` recommends a separate Xiaohongshu connector
  service behind the existing `custom` content provider boundary, not direct
  Spider_XHS integration inside NestJS.
- `packages/web-frontend/app/workbench/outline-workspace.tsx` already preserves
  previous outline batches and keeps the outline UI compact.

## Premises

1. The first implementation should improve the main creator loop, not build a
   full Xiaohongshu operations platform.
2. The NestJS backend should call a stable custom connector contract and should
   not import Spider_XHS or XHS_ALL_IN_ONE directly.
3. The UI should show a compact research summary, not raw copied note content.
4. The research record should be persisted per conversation so the workspace can
   restore the visible state and the backend can debug provider behavior.
5. TikHub configuration stays available for existing import flows, but this new
   research path prefers `custom` and does not silently fall back to TikHub.

## Approaches Considered

### Approach A: Minimal Wiring

Reuse current manual reference import and outline helper paths with small UI
changes. This ships quickly, but it does not create an automatic keyword search,
durable research record, or reliable source-backed outline pipeline.

Effort: S
Risk: Medium
Completeness: 5/10

### Approach B: Product Core

Add a conversation-scoped research run, custom connector keyword-search contract,
agent research helpers, backend `/xhs-analysis/research/outlines`, and frontend
research summary UI. This is the selected path for the first implementation.

Effort: M
Risk: Medium
Completeness: 9/10

### Approach C: Full Connector Platform

Create `packages/xhs-connector` as a standalone connector platform with account
authorization, Cookie storage, SDK adapters, search tasks, content library,
health checks, and future publishing support. This is the right long-term
direction, but too large for the first research-backed outline increment.

Effort: XL
Risk: High
Completeness: 10/10

## Recommended Approach

Use Approach B now. It directly matches the target user flow while preserving the
service boundary recommended by `docs/xhs-connector-analysis.md`.

Approach C is explicitly deferred and recorded as a future iteration. It should
not be mixed into the first implementation unless the user separately asks to
build the connector platform.

## Selected First-Version Scope

- Add agent pure helpers:
  - `buildXhsSearchKeywords`
  - `analyzeXhsPopularSamples`
  - `buildXhsResearchBackedOutlines`
- Add Prisma model `XhsResearchRun` related to `Conversation`.
- Add connector contract:
  - `POST /xhs/posts/search`
  - `Authorization: Bearer <apiKey>`
  - request `{ keyword, limit, sort: "popular" }`
  - response normalized from `data.posts`, `data.items`, or `data.notes`
- Add protected backend endpoint:
  - `POST /xhs-analysis/research/outlines`
- Make the endpoint:
  - verify conversation ownership
  - generate 3-5 keywords in quick mode and 5-8 in deep mode
  - query custom provider with controlled concurrency
  - normalize, deduplicate, rank, and analyze popular samples
  - persist `XhsResearchRun`
  - create a new `OutlineBatch`
  - return research summary plus three outlines
- Update frontend API types and workbench flow.
- Show compact summary above outline candidates:
  - keywords
  - analyzed sample count
  - confidence and sample quality
  - warnings and failed keywords
  - hook patterns
  - outline patterns
  - tag directions
  - content angles
  - avoid patterns
  - compliant source transparency through representative source metadata, not
    raw note content
- Keep previous outline batches when the user clicks `换一批`.
- Support `quick | deep` at the backend DTO/data layer, but keep the first
  frontend experience fixed to `quick` to avoid adding UI mode choice too early.

## Not In Scope

- Building `packages/xhs-connector` in this increment.
- Cookie account management.
- Connector health dashboard.
- Background task queue.
- Global content library.
- Raw note list selection UI.
- Full note body, image, or copied raw payload display in the frontend.
- First-version frontend quick/deep mode switch.
- Direct Xiaohongshu publishing.
- Automatic fallback from `custom` to TikHub.

## CEO Review Scope Decisions

Selected mode: Selective Expansion.

Accepted into first-version scope:

- Research confidence and warnings in the compact summary. The backend should
  expose sample quality, partial provider failure, low-sample warnings, and
  failed keywords so users and developers can judge whether the generated
  outlines are trustworthy.
- Low sample counts should not hard-block outline generation. If the connector
  returns weak or sparse samples, RedNote should still generate editable outlines
  with a clear low-confidence warning, because the user can review, edit, and
  choose whether the result is usable.
- The primary user value is the final usable result, not a statistically
  complete research report. Sparse samples are acceptable as supporting signal as
  long as the app is honest about confidence and keeps the generated outlines
  editable.
- Compliance-aware source transparency. The UI may show source metadata such as
  note ID/URL, truncated title, interaction summary, and match reason. It must
  not display full copied note content, raw payloads, or source images by
  default.

Deferred:

- Frontend deep research mode switch. The backend and persisted data should
  support `quick | deep`, but the first user-facing workflow should stay fixed to
  `quick`. A deep mode entry can be added after latency and provider reliability
  are understood.

## Future Iteration: Approach C

Approach C should become a separate platformization track after the research
outline pipeline proves useful.

Recommended later scope:

- `packages/xhs-connector` standalone service.
- Internal API key authentication.
- PC Cookie import and validation.
- Encrypted Cookie storage.
- Account status: active, expired, invalid.
- Adapter layer around Spider_XHS.
- Search notes, import note, import account, comments, and user notes.
- Rate limiting, retry, timeout, and explicit 429/frequency-control handling.
- Task records for crawl/search/import jobs.
- Optional content library and media cache.
- Health checks surfaced in admin content provider configuration.

This future work should be its own design/spec because it has a different risk
profile: account security, platform compliance, private protocol drift, and
operational maintenance.

## Success Criteria

1. A user can enter one idea and generate three research-backed outline
   candidates.
2. The backend calls the configured `custom` connector search endpoint with the
   configured API key.
3. The frontend shows a compact research summary without rendering raw full note
   content.
4. Old outline batches remain visible after generating another batch.
5. Opening an existing conversation restores the latest visible research summary.
6. Missing custom provider configuration produces a clear setup error.
7. Partial keyword failures can complete with warnings when enough samples remain.
8. Low sample counts still return editable outline candidates, but the research
   summary clearly marks low confidence and explains why.
9. Provider setup failures do not masquerade as research results; they return a
   clear setup or connector error.
