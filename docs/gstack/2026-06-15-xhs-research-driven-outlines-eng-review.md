## Data Flow With Shadow Paths

```text
idea
  ├─ missing/empty -> BadRequest: "先写一句主题，再生成新的大纲。"
  ├─ valid -> keyword plan
  │   ├─ no custom provider -> BadRequest: "请先在后台配置小红书连接器。"
  │   └─ custom provider configured
  │       ├─ some keyword searches fail -> continue, warnings[]
  │       ├─ sparse samples -> generate outlines, low-confidence warning
  │       ├─ zero usable samples from valid connector response
  │       │   -> generate editable fallback outlines, mark fallback/no-sample
  │       └─ connector setup/network failure for all calls
  │           -> clear connector error, no fake research-backed label
  └─ persisted transaction
      ├─ success -> research run + outline batch + selected outline
      └─ DB failure -> rollback all writes
```

## Data Model Plan

Add `XhsResearchRun`:

```prisma
model XhsResearchRun {
  id               String   @id @default(cuid())
  conversationId   String
  idea             String
  mode             String
  providerType     String
  providerEndpoint String?
  keywords         String
  sampleCount      Int
  samples          String
  analysis         String
  summary          String
  status           String
  errorMessage     String?
  createdAt        DateTime @default(now())
  updatedAt        DateTime @updatedAt

  conversation Conversation @relation(fields: [conversationId], references: [id], onDelete: Cascade)

  @@index([conversationId, createdAt])
}
```

Use status values:

- `completed`
- `completed_with_warning`
- `fallback_no_samples`
- `failed`

`fallback_no_samples` means the user still receives editable outlines, but the
frontend must not present them as strongly research-backed.

## Error And Rescue Registry

| Codepath | Failure | Rescue action | User sees | Test |
|---|---|---|---|---|
| `researchOutlines` | empty idea | throw BadRequest | prompt to enter idea | backend spec |
| `ensureOwnedConversation` | not owner | throw NotFound | no access | backend spec |
| `getRuntimeConfig("custom")` | disabled/missing key | throw BadRequest with connector setup copy | setup error | backend spec |
| connector search | one keyword timeout/500 | record failed keyword, continue | warning, outlines still generated | backend spec |
| connector search | all calls network/setup fail | fail connector step, no fake research label | connector error | backend spec |
| connector search | valid empty result | fallback outlines with no-sample warning | editable low-confidence result | backend spec |
| response normalization | malformed shape | per-keyword warning | warning if enough/fallback if none | backend spec |
| transaction | DB write fails | rollback research and outline writes | generic failure | backend spec with mocked throw |
| frontend request | double click | disable while generating | single in-flight request | frontend test |
| restore conversation | missing research summary | render no summary | normal workspace restore | frontend test |

## Security And Compliance

- Endpoint must use `JwtAuthGuard`.
- Conversation ownership check is mandatory before provider calls.
- Provider API key stays server-side through `AdminContentProvidersService`.
- Logs must not include API keys, raw note bodies, or full raw provider payloads.
- UI must not show full source note content or source images by default.
- Store raw/standardized samples only as backend trace data.
- `standoutSamples` should contain metadata and match reasons, not copied content.

## Performance

- Quick mode: 3-5 keywords, 5 results each.
- Deep mode: supported in backend shape only, 5-8 keywords, 10 results each.
- First frontend release always sends `quick`.
- Use controlled concurrency of 2-3 provider requests.
- Use per-keyword timeout. A timed-out keyword becomes a warning unless all
  keywords fail due connector errors.
- Synchronous first version is acceptable because quick mode caps total work.

## Test Plan

### Agent

- `buildXhsSearchKeywords` returns 3-5 keywords in quick mode and 5-8 in deep
  mode.
- `analyzeXhsPopularSamples` deduplicates by note ID/URL/title fallback.
- Ranking uses engagement weight: likes + collects * 1.4 + comments * 1.8 +
  shares * 1.6.
- Sparse samples return low-confidence warnings.
- `standoutSamples` does not include full copied content.
- `buildXhsResearchBackedOutlines` returns three editable candidates.

### Backend

- Endpoint requires authentication.
- Conversation ownership is enforced.
- Missing custom provider returns setup error.
- Request uses `{baseUrl}/xhs/posts/search` and configured API key.
- Partial keyword failures create `completed_with_warning`.
- Valid empty result creates fallback outlines with low-confidence warning.
- Connector setup/network failure for all calls returns connector error.
- Transaction creates research run and outline batch together.
- Opening a conversation returns latest visible research summary.

### Frontend

- `api.xhs.researchOutlines` targets `/xhs-analysis/research/outlines`.
- Workbench uses the research endpoint for outline generation.
- Research summary renders keywords, confidence, warnings, patterns, and source
  metadata without raw note bodies.
- `换一批` keeps old batches and updates latest research summary.
- Restored conversations show latest research summary when present.
- Double-click while generating does not create duplicate requests.
